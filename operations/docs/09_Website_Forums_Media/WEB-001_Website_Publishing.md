# WEB-001 — Publishing Website Updates

> New to staff work? See the [Plain-language glossary](../00_PLAIN_LANGUAGE_GLOSSARY.md) for technical terms.

## 1. Purpose

Publish accurate website updates with the correct review and approval.

## 2. Scope

Homepage, blog/dev logs, rules, terms, privacy, staff applications, guides, store pages, lore pages, navigation, embeds, and other Eldenstead website content.

## 3. Who does what

- **Author:** Drafts and verifies content.
- **Editor:** Reviews clarity, grammar, links, and consistency.
- **Website administrator:** Implements and tests safely.
- **Founder:** Approves legal/policy/store claims, release dates, and major announcements.
- **Lore/content owners:** Approve official or embargoed material.

## 4. Procedure

### 4.1 Content brief
Record audience, goal, page/post type, required links/assets, publication date, approval needs, and whether the content is public, draft, or embargoed.

### 4.2 Drafting standards
- Use human, direct language consistent with Eldenstead.
- Verify server links: `eldenstead.com`, `eldenstead.com/store`, `discord.eldenstead.com`, and `map.eldenstead.com`.
- Do not claim features, dates, refunds, compatibility, or staffing terms that are not approved.
- Clearly distinguish work-in-progress from available features.
- Add meaningful headings, short paragraphs, and alt text for important images.

### 4.3 Technical preparation
1. Save the current page/source before editing.
2. Draft outside the live editor for large changes.
3. Sanitize embedded HTML/iframes and use trusted video sources.
4. Avoid exposing admin routes, server file paths, tokens, or debug output.
5. Preview desktop and mobile layouts.

### 4.4 Review checklist
Check title, description/meta text, author/date, links, images, video ratio, spelling, formatting, accessibility, permissions, official lore, spoiler status, and call to action.

### 4.5 Publish
Use the correct category/visibility and verify the public URL in a logged-out/private session. Test navigation, embeds, forms, Discord/store/map links, and page source for unintended text.

### 4.6 Updates and corrections
For factual updates, change the page and record the revision date. For material policy/terms/privacy changes, preserve prior versions and state effective dates. Correct broken content promptly and avoid deleting historical announcements without reason.

### 4.7 Emergency rollback
Restore the saved version or remove publication if content exposes secrets/private data, creates a security risk, or materially misstates policy/payment terms. Preserve evidence and escalate.

## 5. When to get senior help

Escalate legal/policy text, refund/payment claims, security disclosures, personal data, unlicensed media, major public promises, or suspected compromise of the website/admin account.

## 6. What to record

- Content brief/draft
- Editor and approval
- Previous page backup
- Publication URL/time
- Link/mobile/logged-out verification
- Revision/correction history

## 7. Final checklist

- [ ] Audience and approval needs identified
- [ ] Links and claims verified
- [ ] Existing page backed up
- [ ] HTML/embeds reviewed
- [ ] Desktop/mobile preview checked
- [ ] Logged-out public view verified
- [ ] Revision/effective date recorded
- [ ] Emergency rollback available
